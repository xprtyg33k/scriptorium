Product Outcome Brief
75 min · Product leads · Guillermo & Chris as reviewers
Artifact Product
The Product Outcome Brief is the foundation. Every other artifact today builds on it. Start from the existing Telehealth proposal — do not reinvent. Distill and clarify. Write for an engineer who has never been in the room.

📝 Brief Components to Complete

  • Problem Statement — What is broken or missing for the user today? One paragraph, no jargon.
  • Target User / Workflow — Who is affected, in what context, doing what task?
  • Desired Business Outcome — What changes for the business if this is built?
  • Success Metric — How will we know it worked? Be specific.
  • Strategic Rationale — Why telehealth, why now?

⚙️ Telehealth Context (Pre-loaded)

  • 5+ distinct product impacts: AI Scribe extensions, Telehealth Connect skill for Care Agent, video wrapper service, payments, e-reg
  • Telehealth plan is mature — top-to-bottom spec exists. Use it.
  • Product framing: today's session covers AI Scribe extensions and the Care Agent skill as primary
  • Video wrapper (Andrew) already prototyped — note as input, not scope
  • Contractors expected within weeks — artifacts need to be POD-ready

💡 Facilitation Prompts (if stuck)

  • "What does a clinic experience today that this removes?"
  • "If a contractor read this tomorrow, could they describe what they are building?"
  • "What is the user doing differently in 6 months because of this?"
  • "What is the single most important outcome from the business side?"
  • "What would tell us in 30 days that we shipped the wrong thing?"

🚩 Common Pitfalls to Avoid

  • Writing features instead of outcomes ("build X" ≠ outcome)
  • Leaving the success metric vague ("improve adoption" is not a metric)
  • Conflating the telehealth user (clinic/patient) with the internal stakeholder
  • Letting technical constraints drive the problem framing
  • Treating "we all know it" as documentation — write it out

11:15 – 12:00

MVP Scope & Explicit Exclusions
45 min · Product leads · Guillermo as scope enforcer
Artifact Critical
The MVP boundary is the single most important output of this session. Explicit exclusions are as valuable as inclusions. An agent given vague scope will gold-plate indefinitely. Explicit exclusions protect velocity.

📦 MVP Brief Components

  • MVP Scope — What is the smallest thing that validates the core outcome?
  • Explicit Exclusions — What are we committing NOT to build in this slice?
  • Assumptions — What must be true for this MVP to work that we haven't verified?
  • Open Questions — What do we not know yet? Name them; don't hide them.
  • Acceptance Signals — How will we know the MVP is "done enough" to validate?

✂️ Scope Guard Questions

  • "Is this in because it delivers value, or because it's easier to include?"
  • "Could we validate the core outcome without this feature?"
  • "What is the one thing that, if it fails, makes the whole thing fail?"
  • "What assumptions are we making about dependencies on other systems?"
  • "What are we deliberately deferring — and do we agree on that deferral?"

🔑 Telehealth-Specific Scope Guidance

  • AI Scribe extensions vs. Care Agent skill — confirm which is slice 1 vs. follow-on
  • Payments and e-reg integrations: document as known future dependencies, not MVP
  • Video wrapper: in scope as existing POC input, not in scope as build work today
  • EMR interaction: the team has this mapped — include as known context, confirm scope
  • Contractor readiness: scope must be stable enough for a new resource to act on

⚡ Experimentation vs. Waterfall Framing

  • Engineering time is now cheap. Code is no longer the long pole. Build to learn.
  • An MVP is a validated experiment, not a commitment to a final architecture
  • Explicit exclusions free the team to move — "not in scope" is not "never"
  • The team can run two prototypes and pick one — capture that intent in the brief
  • Write the MVP brief as an input to an AI agent: it must be unambiguous

⚠️ Watch-Outs

  • Telehealth has 5+ product impacts — resist the pull to define all of them today
  • Complexity is real, but that's not an argument for delaying the first slice
  • Do not let "open questions" become an excuse to defer decisions that can be made right now
  • Exclusions must be explicit and written — verbal agreement is not an artifact

✅ Exit Criteria

  • MVP scope boundary is written and agreed
  • Explicit exclusions are listed (minimum 3)
  • Assumptions are documented
  • Open questions are named with tentative owners
  • Acceptance signals are defined

Candidate Execution Slices
60 min · Product + Chris + Guillermo · Frame outcomes, not tasks
Artifact Strategy
This is where we break the front-heavy product process into smaller, agile-friendly slices. Each slice is a product outcome — not an engineering task. Each slice must be small enough to build, validate, and learn from before committing to the next. This is where Chris's insight lands: early experimentation, market signal, and fast follows.

✂️ What a Good Execution Slice Looks Like

  • Stated as a product outcome ("clinics can initiate a telehealth visit from within the EMR"), not an engineering task
  • Small enough to build, ship, and get signal from within one sprint cycle
  • Has an explicit validation gate: "what must be true before we start slice 2?"
  • Interdependencies with other slices are identified and sequenced
  • Throwaway code is acceptable — learning is the output of early slices

🧩 Telehealth Candidate Slices (Starter)

  • Slice 1 (AI Scribe): Core telehealth features extension to existing scribe product
  • Slice 2 (Care Agent): Telehealth Connect skill for Care Agent — zero to one
  • Slice 3 (Video): Integrate Andrew's video wrapper into the workflow
  • Deferred: Payments, e-reg, and cross-system touch points
  • Validate ordering today — does the team agree on sequencing?

💡 Facilitation: Building the Slice Map

  • Start with slice 1 only — resist defining all slices in detail upfront
  • For each slice: what is the user able to do that they couldn't do before?
  • "What must we validate before an AI agent or developer can safely act on this?"
  • Name the validation gate for each slice — this prevents runaway scope
  • Capture fast-follow candidates in a "parking lot" — don't lose them, but defer them

⚡ Shifting the Mindset

  • Old model: don't start coding until 100% sure — throwaway code is waste
  • New model: code is cheap — run experiments, get market signal, learn fast
  • Two prototypes, pick the better one — that's the new norm
  • Engineering joins earlier in the process precisely because iteration is cheap
  • The blocker is now: how fast can Product feed requirements to the agents?
