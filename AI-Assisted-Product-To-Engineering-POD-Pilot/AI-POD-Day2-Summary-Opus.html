Lessons Learned

Day 2 surfaced lessons about the process itself—not just the telehealth content. These are the patterns to carry forward and the anti-patterns to avoid.

Read Before Reacting Works

The silent cold read created higher-quality feedback than any live walkthrough could. When reviewers consume artifacts before discussing, they identify real gaps instead of reacting to presentation style. This should be the standard for every handoff.

Context Loading Is The Critical Path

The prior POC lesson was confirmed: coding agents stall when the context funnel is incomplete. The packet must be loadable into an AI workspace to be truly useful. Integration context and system architecture documentation are first-class requirements, not afterthoughts.

Outcome Language Enables, Prescription Constrains

PRD sections written in outcome language produced faster engineering action. Sections that prescribed implementation details (endpoints, schemas, transport) slowed the review because Engineering had to negotiate boundaries instead of building. Product should specify the what, never the how.

Events Are First-Class Requirements

Treating business events and operational metrics as product requirements rather than after-the-fact telemetry eliminates a major class of post-launch data gaps. The event inventory must be part of the standard artifact packet.

Process Incompressibility Requires Slicing

The product process cannot be compressed the way code can. The solution is not to rush requirements—it is to break the front-heavy process into smaller, outcome-based slices so engineering experimentation can begin earlier. Early spikes and POCs are not waste; they are signal.

Shared Memory Is Non-Negotiable

Day 1 started with an access friction issue that proved the point: artifacts that live in personal drives or disconnected folders cannot be consumed by people or agents. A confirmed, accessible shared location must be the first decision in any initiative.

Confidence Signals

What created confidence in the packet.

Integration Context Gaps

Where Engineering had to infer rather than read.

Specifics

Care Agent skill dependencies (carrier event schema, EMR event types for visit initiation) were insufficiently detailed. The systems inventory was partial. Engineering needed to ask clarifying questions about system interactions.

Implication

Add a pre-loaded systems inventory step before the handoff. Architecture should contribute integration context before the validation session, not during it.

Over-Prescription Concerns

Where Product crossed into Engineering territory.

Specifics

Some PRD sections included transport mechanisms (Kafka topics), schema suggestions, and endpoint naming that constrained Engineering choices. These sections created friction because Engineering had to determine whether they were requirements or suggestions.

Implication

Enforce the lane rule: PRDs must not specify implementation mechanisms. Product specifies the event and the data; Engineering determines the transport and schema.

Validation Gate Clarity

How Engineering knows when a slice is done.

Specifics

Slice 1 validation gate was stated as a quality indicator rather than a binary observable event. Engineering needs a specific, verifiable signal to know when to move to the next slice.

Implication

Every slice must have a binary go/no-go gate expressed as a measurable event, not a directional sentiment.

Experimentation Model

Engineering's posture toward early prototyping.

Specifics

Engineering confirmed willingness and capacity to run early experiments and POCs before full product definition is complete. The shift from "throwaway code is waste" to "prototypes are signal" was embraced. Agents can code spikes while Product iterates on definition.

Implication

The process must define a formal experimentation window where Engineering acts on partial context to generate learning, not just production code.

Product Process: Pre And Post Onsite

The onsite exposed and addressed the gap between the legacy handoff model and the AI-native operating model that accelerated delivery demands.

Pre-Onsite State

  • No consistent artifact packet structure across initiatives.
  • Product handoff was verbal or scattered across documents in personal drives.
  • Engineering received tickets from a single product owner rather than shared context.
  • No formal validation step existed between Product definition and Engineering execution.
  • Events and metrics were treated as telemetry, not requirements.
  • Engineers were positioned as ticket-takers waiting for fully-defined work.
  • Integration context was discovered during sprints, not loaded before them.
  • No shared AI-accessible memory location for artifacts.

Post-Onsite State

  • Defined minimum viable artifact packet: strategy, PRDs, event inventory, acceptance criteria, open questions, handoff brief.
  • Shared, AI-accessible artifact location confirmed and named.
  • Validation step formalized: blind cold read before engineering acts.
  • Ownership lanes codified with clear boundaries.
  • Events and metrics elevated to first-class product requirements.
  • Engineers positioned as active contributors with context, not passive ticket-takers.
  • Integration context pre-load identified as a required step.
  • AI workspace pattern documented for reproducible artifact generation.

Proposed Engineering Process

The validated product-to-engineering handoff pattern, codified as a repeatable process for any initiative.

1. Signal Product gathers customer research, market signals, strategy decisions, and business context. Output: product brief with problem, users, and business rationale.
2. Context Load Product creates artifact packet using AI-native tooling. Architecture contributes systems inventory and integration context. Output: PRDs, event inventory, acceptance criteria, open questions.
3. Validate Engineering and Architecture perform blind cold read. Feedback categorized by artifact and failure mode. Output: comprehension score, gap list, artifact change requests.
4. Execute Engineering generates TRD from packet plus architecture standards. Work items, spikes, QA criteria, and agent prompts produced. Early experiments run in parallel.
5. Memory Decisions, artifacts, and lessons written to shared memory location. Durable, AI-accessible, and findable for the next initiative.
Artifact Packet

Minimum Viable Artifact Packet

Every initiative handoff must include the following artifacts before Engineering validation begins:

  • Solution Strategy: vision, target customers, product relationships, roadmap intent, and guardrails.
  • Module PRDs: problem, users, outcomes, scope, exclusions, events, metrics, and acceptance criteria.
  • Event And Metrics Inventory: workflow events, operational metrics, reporting data, and product usage signals.
  • Systems Inventory: known integrations, dependencies, interaction types, and owners.
  • Open Questions: categorized as blocking, confidence-building, or future-phase, with owners and timelines.
  • Handoff Brief: what each artifact is, why it exists, and what feedback is requested.
Channel Flow

Communication Process And Channels

The handoff is not a single event—it is a structured flow with defined communication points:

  • Artifact packet delivered to shared location with cover note explaining read order and feedback expectations.
  • Silent cold read period: Engineering consumes without live narration.
  • Structured feedback session: concerns mapped to specific artifacts, sections, and failure modes.
  • Lane adjustment discussion: resolve boundary questions before execution begins.
  • Action items with owners and dates documented in shared memory.
  • Async updates through the shared artifact location—no verbal-only decisions.
Guard Rails

Process Guardrails

The following rules protect the process from the failure modes surfaced during the pilot:

  • PRDs must not prescribe implementation mechanisms—outcome language only.
  • Every slice must have a binary validation gate, not a directional quality indicator.
  • Integration context must be pre-loaded before validation, not discovered during sprints.
  • No verbal-only decisions: if it is not written in the artifact, it does not exist.
  • Experimentation is not waste: early spikes and POCs are an explicit part of the process.
  • Artifacts must be AI-accessible: shared folder with consistent naming, not personal drives.
