Day 2 Process Summary

Day 2 confirmed that the product-to-engineering handoff pattern is valuable, but the artifact packet must become more explicit before it can reliably support new engineers, architects, JIRA generation, and AI-assisted delivery. The strongest signal was not that the documents were wrong; it was that hidden institutional context surfaced immediately when engineers read them cold.

Reviewers consumed the packet with limited context, then reported what they understood and where they needed clarification.

  • Reading time was treated as part of the process, not a pre-work assumption.

The team compared a more complete PRD with a condensed MVP artifact pack and learned that shorter is not automatically clearer.

  • Both formats still needed definitions and stronger boundaries.
  • The MVP pack is useful as an entry point, but not a substitute for needed context.
  • A built-in review guide should become standard.

Engineering surfaced the exact gaps that usually become sprint churn: unclear terms, contradictory statements, hidden dependencies, and implementation-coded phrasing.

  • Payment status and registration system were under-defined.

LanesClarified

The team reinforced that Product owns the plain-language product ask, while Engineering and Architecture own the TRD and implementation approach.

  • Product should describe outcomes and business rules.
  • Engineering should decide APIs, services, events, schemas, and implementation mechanisms.
  • Architecture standards and system context should be referenced, not duplicated everywhere.
ChannelsExposed

  • Meeting chat can distribute the link, but not serve as memory.
  • Inline document comments are useful for artifact-specific feedback.

Day 2 Validation Findings

The gaps were mostly process and artifact-design issues, not product-idea issues.

Comprehension Pass Directional

The Broad Ask Was Understandable

Cold readers could summarize the intended workflow at a high level: a patient-side preparation/check process marks a patient ready, then the patient enters a telehealth/provider flow.

  • This indicates the packet had enough product signal to orient reviewers.
  • The gaps emerged after the high-level understanding, mostly at decision and dependency boundaries.
Definitions Fail Critical

Implicit Vocabulary Became The Main Failure Mode

Terms like skill, Care Agent, Telehealth Connect, registration system, payment status, token, writeback, eligible, ready, encounter, and AI Scribe were not explicit enough for a cold reader.

  • Definitions should live in the packet front matter or glossary.
  • Terms should link to their source artifact or canonical system reference.
  • Cold readers should not have to know Experity history to understand the artifact.
Contradiction Fail Fix

Conflicting Requirements Confused Both Humans And AI

The state-attestation language gave conflicting signals about whether the patient could proceed. The team correctly recognized this would confuse an AI agent as much as a human engineer.

  • Every gate should state whether it blocks readiness, only records data, or flags an exception.
  • Future behavior should be labeled as future behavior, not mixed into MVP rules.
Artifacts Partial Improve

The README Was Useful, But The Packet Needs A Standard

The front-door README helped orient the session, but reviewers still needed clearer artifact purpose, order, audience, and cross-links.

  • Each artifact should say who it is for and what decision it supports.
  • Open questions should be categorized by audience and implementation impact.
  • References should point to exact sections, not just whole documents.
Language Partial Guardrail

Product Language Must Be Plain, Precise, And Outcome-Based

Words that sound harmless to Product can become implementation constraints to Engineering. "Processed through AI Scribe" was interpreted as a technical claim that could invalidate the current proof-of-concept path.

  • Create a "do not use" list for generated PRDs: API, Kafka, endpoint, pipeline, processed, streamed, token, table, schema, unless the technical artifact owns it.
  • Replace implementation wording with product outcomes and user-visible behavior.
Visuals Partial Needed

Design And Workflow References Are Part Of The Packet

Reviewers asked for workflow diagrams, sequence-like user flows, screenshots, mocks, and Figma links. These are not implementation design; they are context that prevents engineers and agents from guessing.

  • PRDs should link to design artifacts rather than embedding every screen.
  • Workflows should show the user interaction and system handoff at product level.

Engineering Comments And Concerns

These are the process-relevant concerns raised by Engineering and Architecture during the cold read.

Definitions Are Required Context

Engineering does not need technical internals in the PRD, but it does need to know what the named systems, modules, users, and states mean.

Concern

Without definitions, reviewers infer from prior knowledge or make incorrect assumptions.

Process Response

Add a glossary, dependency map, and exact source links to the packet standard.

PRD Should Not Dictate The How

Engineering pushed back on API, Kafka, pipeline, processing, token, and similar terms in product artifacts.

Concern

Technical wording can unintentionally constrain architecture or cause engineers to throw away valid implementation options.

Process Response

Move implementation details to TRD, architecture review, or implementation plan.

Visual Context Is Not Optional

Workflows, Figma links, screenshots, and user flow references are useful handoff artifacts.

Concern

Text-only requirements can still leave too much room for a developer or AI agent to invent the experience.

Process Response

Make visual and workflow references part of the minimum packet when UI or workflow behavior matters.

Success Criteria Need Tiers

Engineering questioned whether metrics in a PRD were real, measurable, and available.

Concern

AI-generated metrics can be superficial or dependent on events that do not exist yet.

Process Response

Separate strategy metrics, functional acceptance criteria, operational telemetry, and TRD performance standards.

Open Questions Need Routing

Some open questions read like product notes, some architecture notes, and some engineering blockers.

Concern

Engineers may ignore open questions if they are not clearly tied to implementation impact.

Process Response

Tag every open question by audience, owner, decision needed, and whether it blocks work.

Process Definition Updates

Day 2 moved the process from concept to a working draft.

1. Product Packet Product prepares strategy context, handoff README, module PRD, glossary, workflow/design links, assumptions, open questions, and acceptance criteria.
2. Cold Read Engineering reads silently, comments in place, and records where the packet answers or fails to answer the standard review questions.
3. Gap Triage Gaps are classified as definition, dependency, contradiction, over-prescription, acceptance criteria, design context, or TRD input.
4. TRD Path Engineering and Architecture create the TRD from the product packet, architecture standards, Architecture Hub, and known system context.
5. Delivery Artifacts Validated packet plus TRD generates JIRA epics/stories, spikes, QA checks, agent prompts, and onboarding context.
Minimum Artifact Packet For The Next Trial

Product-Owned

  • Handoff README with reading order, artifact purpose, audience, and review questions.
  • Product or solution strategy with durable context, customer signal, roadmap intent, and business constraints.
  • Module PRD with plain-language outcomes, users, scope, non-goals, assumptions, business rules, dependencies, and functional acceptance criteria.
  • Glossary and dependency map for named systems, states, modules, and product concepts.
  • Workflow and design references: Figma, screenshots, user flows, workflow diagrams, sequence-like product flows.

Engineering-Owned

  • TRD or architecture implementation brief.
  • Architecture review notes and standards references.
  • System contracts, interface decisions, event implementation, service boundaries, data model details, and compliance implementation approach.
  • JIRA delivery breakdown, spikes, QA scenarios, and AI-agent context bundle.
Communication Process And Channels

Durable Channels

  • Shared product intelligence hub or wiki for final artifacts, glossary, and cross-links.
  • Inline document comments for artifact-specific review feedback and resolution history.
  • Architecture Hub for system context, standards, and technical references.
  • JIRA only after packet validation and TRD direction are ready enough to generate work.

Transient Channels

  • Meeting chat can distribute the handoff README link during the session.
  • Teams/chat can coordinate scheduling and quick follow-up, but should not become the source of truth.
  • Session notes should be written back into the durable artifact location before the work moves forward.

Lessons Learned

Day 2 turned the packet into a practical test surface. The main lessons are repeatable beyond this product.

Cold Read Beats Assumption Review

Readers with limited domain context surfaced hidden vocabulary and dependency gaps that the Day 1 team could not see from inside the work.

Ambiguity Has A Cost

Contradictory or under-defined statements create immediate engineering churn and can cause AI agents to choose the wrong interpretation.

Plain Language Is A Discipline

Product can be precise without being technical. The right level is understandable by someone outside R&D, while still specific enough for Engineering.

Artifact Length Is Not The Metric

The shorter MVP pack still carried many of the same questions. The real metric is whether the reader can answer the standard review questions.

TRD Is The Technical Bridge

The PRD should not become the technical design. The TRD is where architecture, system contracts, and implementation details belong.

References Need Precision

Linking to a document is not enough. The packet needs exact section references or repeated essential context where the reader needs it.

Action Items And Follow-Up Exercises

These are the follow-ups that move Day 2 learning into a repeatable process.

Revise PRD Template

Add required glossary, dependency map, plain-language outcome rules, cross-linking expectations, and review guide questions.

  • Remove technical trigger words from generated Product text.
  • Separate functional acceptance from strategic metrics and technical performance.

Create Handoff README Standard

Make the README the front door for each packet: purpose, artifact order, audience, how to comment, and pass/fail questions.

  • Include "what should you be able to answer after reading?"
  • Include "what questions should be routed to Product, Architecture, or Engineering?"

Define TRD Handoff

Document when the TRD is created, who owns it, and what inputs it consumes from the validated product packet.

  • Use architecture standards and Architecture Hub as the technical context layer.
  • Keep implementation details out of the PRD unless they are non-negotiable constraints.

Run A JIRA And Agent Dry Run

Use the revised packet to generate sample epics, stories, spikes, QA checks, and AI-agent prompts.

  • Mark every generated work item with the source artifact section.
  • Use weak output as evidence of missing context.

Repeat Cold-Read Validation

Use at least one engineer or architect with limited domain context on the next trial.

  • Time the read.
  • Capture first-pass confusion before discussion.
  • Classify gaps by failure mode.

Schedule Process Retrospective

Hold a follow-up session after artifact updates to finalize the process standard.

  • Review which packet components were most useful.
  • Decide source-of-truth location and naming/versioning rules.
  • Publish the repeatable playbook.

Recommended report-out: Day 2 proved that the process should be adopted, but not yet frozen. The next version should preserve the cold-read validation, strengthen the packet standard, and formally add Engineering-owned TRD generation as the bridge from product clarity to implementation autonomy.
