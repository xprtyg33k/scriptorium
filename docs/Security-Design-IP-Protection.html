Board Mandate Response v4.0

Developer Velocity

Addressing Board-level directives to neutralize IP exfiltration while preserving elite engineering performance via CrowdStrike, Rapid7, and GitHub EMU.

The Executive Vision

Following identified intellectual property breaches—specifically the duplication of repositories by external contractors—the Board has mandated a definitive security pivot. We move from User-based trust (which failed) to Environment-based control.

This is not a blanket lockdown. Our core team (FTEs, Architects) must move fast. We achieve this through a "Tiered Persona" model: an "Iron Cage" for contractors and a "Trusted Corridor" for internal staff.

The Strategy: Tiered Personas

We move the security boundary from User-based trust (which failed) to Environment-based control. Each persona is governed by a distinct combination of identity model, working environment, and security posture.

Persona Comparison Matrix

Tier Persona Identity Model Working Environment Security Posture
Tier 1 Contractors GitHub EMU (Managed) Secure VDI / CDE (Coder) Iron Cage — Zero local data, blocked clipboard
Tier 2 Trusted FTEs SSO-Linked Personal Direct (Local / VPN) Accountable Velocity — Full tool flexibility + logging

Why Tiering? Senior technical resources require the ability to fork repos, contribute to Open Source, and experiment with external AI tools (e.g., Claude Code). Forcing them into a high-latency, locked-down VDI would significantly degrade velocity. By tiering, we close the "front door" for the highest-risk group (contractors) while maintaining a "monitored hallway" for employees.

Technical Deep Dive

Identity Layer: GitHub Enterprise Managed Users (EMU)

EMU is the architectural cornerstone of Tier 1 control. Unlike standard GitHub organizations where contractors use personal accounts you cannot govern, EMU means the account itself is corporate property — provisioned by us, revoked by us, and technically constrained at the platform level.

Step-by-Step Provisioning

  1. Create a net-new GitHub Enterprise account. A GitHub Enterprise Managed Users instance cannot be created by converting an existing organization. This must be a brand-new enterprise account — contact your GitHub account team to initiate. The handle will be experity-emu.
  2. Connect Okta as the Identity Provider via SAML + SCIM. In the EMU Enterprise settings, navigate to Authentication Security → SAML Single Sign-On and configure the Okta SAML application. Then enable SCIM provisioning under Provisioning using Okta's GitHub EMU connector app. Map the contractor Okta group to the EMU organization. From this point, contractors are auto-provisioned with usernames formatted as jsmith_experity — an account that exists only within our namespace.
  3. Set Enterprise-level governance policies. In the Enterprise policy admin panel, enforce the following — these cannot be overridden at the org or repo level:
    • Disable Repository Forking — prevents any fork, even within the org.
    • Restrict Email Notifications to Verified Domains — blocks exfil via notification emails.
    • Disable Personal Access Token (classic) creation — force fine-grained PATs with admin approval.
    • Block public repository creation — all repos default to private.
  4. Mirror existing repositories into the new org. Use git clone --mirror to duplicate critical repos into experity-emu. Do not grant contractors access to the original org during this transition. After validation, revoke all legacy contractor access to the primary org.
  5. Inject the Tenant Restriction header at the VDI network proxy. Configure the VDI's egress proxy to insert the sec-GitHub-allowed-enterprise: experity-emu HTTP header on all outbound requests to github.com. This instructs GitHub's own authentication servers to reject any login attempt for an account that does not belong to the experity-emu enterprise — even if the contractor knows their personal credentials.

VDI Network Security & Egress Rules

These rules are applied at the VDI subnet's egress firewall/proxy — not on the individual workspaces. All traffic from Coder workspace pods must traverse this layer. The GitHub IP Allowlist acts as a second, independent "geofence": only requests arriving from the VDI cluster's egress IP range are accepted by GitHub at all.

Firewall Ruleset (VDI Subnet Egress)

Protocol / Port Destination Action Rationale
HTTPS (443) github.com/experity-emu/* ALLOW Approved org only — all other GitHub paths denied
HTTPS (443) *.okta.com ALLOW SSO / SCIM identity provider
HTTPS (443) Approved package registries (npm, PyPI, NuGet) ALLOW Dependency resolution for build tooling
SSH (22) Internet (*) DENY Eliminates uninspectable git push via SSH — all git traffic must go over HTTPS where it can be inspected
HTTPS (443) Personal github.com accounts / namespaces DENY Blocks push to personal account even if SSH is unavailable
HTTPS (443) gitlab.com, bitbucket.org, codeberg.org DENY Closes lateral exfiltration path to competing code forges
All other outbound Internet (*) DENY Default-deny posture — unlisted destinations are blocked

VDI Protocol Lockdown (RDP / HTML5 Session Settings)

  • Disable Clipboard Redirection (Remote → Local). In the Coder/RDP session policy, set DisableClipboardRedirection=true. A contractor cannot copy code out of the VDI session and paste it locally, even if their own machine is running.
  • Disable Drive / USB Mapping. Set DisableDriveMapping=true and block USB mass storage device class in the VDI host policy. Prevents mounting the contractor's local filesystem inside the remote session for drag-and-drop exfiltration.
  • Disable Printing Redirection. Block the redirect of the local printer into the VDI session — a lesser-known but real exfiltration vector via PDF printing.

Critical Bypass Scenario: Re-init & Personal Push

A sophisticated contractor could attempt to delete the .git folder within the VDI workspace, run git init, add their personal GitHub as a remote, and push. This circumvents EMU identity controls because git at the protocol level doesn't enforce account ownership. This architecture defeats the attack at every layer:

  1. Network blocks the push destination. The DENY rule for personal github.com paths rejects the HTTPS push at the firewall before it reaches GitHub.
  2. SSH is fully blocked. The attacker cannot fall back to git remote add origin git@github.com:personal/repo.git because Port 22 is denied to all destinations.
  3. Tenant restriction header rejects authentication. If they attempt to authenticate via browser, the sec-GitHub-allowed-enterprise header causes GitHub's auth servers to reject the personal session.
  4. CrowdStrike detects the anomalous git sequence. A custom IOA fires on git init followed by git remote add with a non-approved URL, generating an alert in real-time before the push can even be attempted.

Endpoint Protection: CrowdStrike Falcon Configuration

Falcon is deployed in two distinct contexts requiring separate sensor groups and prevention policies. Configuration changes are made in the Falcon Console → Endpoint Security → Prevention Policies.

Deployment: VDI Kubernetes Nodes (Coder CDE)

  • Deploy the Falcon sensor as a DaemonSet on the Kubernetes nodes hosting Coder workspace pods (not inside the pods themselves). This ensures coverage at the host level regardless of which workspace is active, and survives workspace restarts.
  • Create a dedicated VDI Node sensor group in the Falcon console. Assign a tailored prevention policy to this group so VDI-specific rules don't bleed into other server or endpoint policies.
Enable (VDI Policy)
  • Suspicious Process Blocking — catches mass-clone scripts launched from a shell
  • Script-Based Execution Monitoring — detects bash/python/PowerShell initiating git operations
  • Lateral Movement Detection — flags unusual network connections originating from git processes
  • File Write Volume Monitoring — alerts on bulk writes to /tmp or home dirs (staging area for exfil)
  • Process Hollowing & Injection Prevention — prevents malicious code injected into a trusted IDE process
  • Full Process Tree Logging — captures complete parent/child chain for every git, curl, wget, scp, rsync, zip, tar invocation
Disable / Restrict (VDI Policy)
  • Do NOT allow local policy overrides — sensor config must be locked to the console-defined policy; no local config files honored
  • Disable USB device class — belt-and-suspenders alongside the RDP drive-mapping lockdown
  • Disable sensor tampering — ensure cs-falconctl cannot be run by the workspace user; enforce via Linux file permissions on the node

Custom IOA (Indicator of Attack) Rules — VDI Group

Navigate to Endpoint Security → Custom IOA Rules and create the following rules assigned to the VDI sensor group:

  1. Git invoked from non-IDE parent process. Rule: Process name = git, Parent process NOT IN (code-server, node, java [IntelliJ]). Severity: High. Action: Alert + kill process. This catches a contractor dropping to a raw shell to run git commands outside the sanctioned IDE.
  2. Re-init followed by unauthorized remote add. Rule: Command line matches git init OR git remote add where the remote URL does not contain github.com/experity-emu. Severity: Critical. Action: Alert + block + isolate host.
  3. Bulk archive creation in working directory. Rule: Process name IN (zip, tar, gzip) operating on paths under /home/* or /workspace/* with file count > 50. Severity: High. Action: Alert.
  4. Network connection from git process to non-approved IP. Rule: Process = git, outbound connection destination NOT IN the approved GitHub/Okta IP ranges. Severity: Critical. Action: Alert + block + isolate host.

Deployment: FTE Laptops (Tier 2)

Enable (FTE Laptop Policy)
  • Behavioral Threat Detection — baseline normal dev behavior, alert on deviations
  • Intelligence-Driven Protection — cloud-delivered IOC feeds
  • Full Process Tree Logging — same as VDI for audit trail parity
  • Device Control — log all USB mass storage events; alert on first-use of any new device
Avoid Over-restriction (FTE Policy)
  • Do NOT apply VDI's process-kill IOA rules — FTEs legitimately run git from terminal
  • Do NOT block non-approved git remotes — FTEs contribute to OSS; apply alert-only posture instead

SIEM & UEBA: Rapid7 InsightIDR Configuration

InsightIDR serves as the unified audit plane — the single place where every identity, network, and endpoint event is correlated. The goal is to make invisible activity visible, and make impossible-to-miss alerts unavoidable.

Step 1: Connect Log Sources

  1. GitHub Enterprise Audit Log. Configure GitHub Audit Log Streaming (under Enterprise Settings → Audit Log → Log Streaming) to stream to an S3 bucket. In InsightIDR, add an Amazon S3 log source pointed at that bucket with the GitHub Audit Log parser. This captures every git.clone, git.push, repo creation, and permission change across the org.
  2. Okta System Log. In InsightIDR, add an Okta log source using the Okta System Log API (OAuth service app). This surfaces contractor authentication events, session starts/ends, MFA failures, and deprovisioning actions.
  3. CrowdStrike Falcon Event Stream. Use the InsightIDR CrowdStrike Falcon native integration (Settings → Log Sources → Cloud Services). Connect via the Falcon Event Streaming API. This brings endpoint detections, IOA alerts, and process telemetry directly into InsightIDR for cross-source correlation.
  4. VDI / Coder Infrastructure Logs. Configure the Kubernetes nodes to forward syslog (or use a Fluent Bit DaemonSet) to InsightIDR's collector. This captures VDI session start/end, pod scheduling, and any system-level events that Falcon doesn't surface.
  5. VDI Subnet Firewall/Proxy Logs. Forward egress firewall deny logs from the VDI network boundary into InsightIDR. Every denied connection attempt — including attempted pushes to personal GitHub — will appear as a log event correlated against the contractor's Okta identity.

Step 2: UEBA Baseline & Settings

  • Enable the Attacker Behavior Analytics (ABA) module. In InsightIDR Settings → Detection Rules, ensure the ABA module is active. This provides pre-built detection logic for data exfiltration, credential abuse, and lateral movement patterns that complement our custom rules.
  • Set a 7-day UEBA baseline period. Before going live, allow 7 days of normal contractor activity to establish behavioral baselines. Anomaly detection thresholds (clone frequency, session duration, active hours) will be calibrated relative to this baseline rather than generic industry defaults.
  • Set log retention to 90 days minimum. In InsightIDR Settings → Data Retention, configure 90 days for all log sources in scope. This satisfies most incident response investigation windows and aligns with typical contractual audit requirements.

Step 3: Custom Alert Rules

Create these as Custom Alerts under Detection Rules → Custom Alerts in InsightIDR. Assign all to the Critical or High priority queue and route to PagerDuty/email:

  1. High-volume clone event. Trigger when a single contractor identity generates more than 5 git.clone events within a 1-hour window (sourced from GitHub Audit Log). This is the direct analog of the original breach behavior.
  2. Push to unauthorized destination. Trigger on any git.push GitHub Audit Log event where the target repository is outside the experity-emu org. Severity: Critical. This should never happen by design; any instance indicates a configuration failure.
  3. Firewall deny on git-related ports from VDI. Trigger when a contractor source IP (VDI range) generates a firewall DENY on Port 22 or a DENY to gitlab.com / personal GitHub paths. This means a bypass was attempted and blocked — confirm the block held and investigate.
  4. Okta deprovisioning lag alert. Trigger when a contractor Okta account remains active more than 4 hours after their engagement end date (requires Okta field for contract end date). Catches off-boarding delays before they become windows of opportunity.
  5. First access from new IP / geolocation. Trigger on a contractor's first-ever Okta authentication from an IP outside the known VDI egress range. This flags sessions initiated from outside the expected environment.
  6. CrowdStrike critical IOA cross-correlation. When a Falcon Critical IOA fires on a VDI node (e.g., the re-init/unauthorized remote rule), automatically correlate it against the current active Okta session to identify the specific contractor identity within seconds.

Step 4: Operational Dashboard

  • Create a dedicated "Contractor Git Activity" dashboard in InsightIDR with the following cards: daily clone volume per contractor identity, active VDI sessions vs. GitHub sessions (should always be 1:1), firewall deny events over time, and a live feed of all CrowdStrike IOA detections from the VDI sensor group.
  • Configure a weekly automated report (InsightIDR → Reporting) covering contractor access summaries, alert counts, and any anomaly investigations. Distribute to the CISO and Engineering leadership as evidence of ongoing compliance with the Board mandate.

Implementation Roadmap

Three sequential phases. Each phase must be validated before the next is activated to avoid locking out legitimate users or breaking existing workflows.

Phase 1: EMU Foundation & Identity Migration

  1. Engage GitHub to provision a new EMU Enterprise account. Submit a request through your GitHub account team. Do not attempt to convert an existing organization — EMU requires a fresh enterprise instance. Estimated provisioning time: 2–3 business days.
  2. Configure Okta SAML + SCIM for the EMU instance. Install the "GitHub Enterprise Managed User" app in Okta. Configure SAML SSO using the metadata GitHub provides. Enable SCIM provisioning and map the Contractors Okta group to the EMU org with the Member role. Validate that a test contractor is auto-provisioned with the _experity suffix.
  3. Enforce Enterprise-level governance policies. In the EMU Enterprise admin panel (Settings → Policies), set: Forking → Disabled, Public repositories → Disabled, Email notifications → Verified domains only, Personal access tokens (classic) → Disabled, Fine-grained PAT approval → Admin required.
  4. Mirror all in-scope repositories. For each repository contractors need access to, run: git clone --mirror https://github.com/experity/<repo>.git && cd <repo>.git && git push --mirror https://github.com/experity-emu/<repo>.git. Set up a short-term mirroring job to keep in sync during the transition period.
  5. Revoke legacy contractor access. Once repos are mirrored and contractors have validated access via EMU, remove their legacy personal-account collaborator invitations from the primary experity org. Verify in the GitHub org audit log that no personal accounts retain access.

Phase 2: "Iron Cage" VDI Deployment & Hardening

  1. Deploy Coder on the internal Kubernetes cluster. Use the official Coder Helm chart. Configure workspaces to use a base image that includes the required IDE (VS Code Server) and language toolchains. Each contractor gets one workspace mapped to their Okta/EMU identity. Workspaces should be ephemeral — destroyed at session end, rebuilt at next login.
  2. Apply the VDI egress firewall ruleset. Configure the subnet-level network policy (or Kubernetes NetworkPolicy + external firewall) with the full ruleset defined in the Technical Deep Dive section: ALLOW approved GitHub org + Okta + package registries; DENY SSH (22), personal GitHub, GitLab, Bitbucket, and all other destinations by default.
  3. Configure GitHub IP Allowlist. In the experity-emu Enterprise Settings → IP Allow List, add only the static egress IP(s) of the VDI Kubernetes cluster's NAT gateway. Enable the allowlist. Verify that accessing the org from any other IP returns a 403.
  4. Inject the tenant restriction header at the VDI proxy. Configure the egress HTTP proxy (Squid, Nginx, or cloud-native equivalent) to add sec-GitHub-allowed-enterprise: experity-emu to all HTTPS requests to github.com. Enable SSL inspection on the proxy so this header is inserted into the inner TLS request, not just the tunnel.
  5. Lock down the VDI session protocol. In the Coder workspace template or RDP broker policy, disable: clipboard Remote→Local, drive/USB mapping, printer redirection, and audio output redirection. Test each control with a contractor-profile test account before go-live.
  6. Deploy CrowdStrike Falcon DaemonSet. Install the Falcon sensor on Kubernetes nodes using the Falcon Operator or DaemonSet manifest provided by your CrowdStrike account team. Assign nodes to the VDI sensor group. Apply the prevention policy and custom IOA rules defined in the Technical Deep Dive. Verify detections fire correctly using a controlled test (run git init from a raw shell, confirm alert appears in Falcon console within 60 seconds).

Phase 3: Employee "Green Lane" & Monitoring Go-Live

  1. Route FTE GitHub traffic through the Corporate VPN. Configure split-tunnel VPN so that all traffic to github.com exits through a defined corporate egress node with a static IP. Add this IP to the primary org's GitHub IP Allowlist (separate from the EMU org's allowlist). FTEs connect to their personal SSO-linked accounts through this path with full tool flexibility.
  2. Validate FTE velocity is unimpaired. Have a senior lead confirm that local git operations, Claude Code, CI/CD pipelines, and IDE integrations all function correctly over the VPN before enforcing the allowlist. Velocity confirmation is a go/no-go gate for activating the IP allowlist restriction on the primary org.
  3. Connect all InsightIDR log sources. Follow the step-by-step log source configuration in the Technical Deep Dive: GitHub Audit Log streaming (S3), Okta System Log, CrowdStrike Falcon Event Stream, VDI syslog, and VDI firewall deny logs. Verify each source is receiving data in InsightIDR's log search within 15 minutes of configuration.
  4. Run the 7-day UEBA baseline period. With all log sources connected but alert rules in audit-only mode, allow 7 days of normal contractor activity to flow. Review baseline metrics daily. After 7 days, promote alert rules to active and configure PagerDuty/email routing.
  5. Create the Contractor Git Activity dashboard and weekly report. Build the InsightIDR operational dashboard and schedule the automated weekly compliance report for distribution to the CISO and Engineering leadership. This is the standing evidence of compliance with the Board mandate.

Summary of Mitigation

This architecture directly addresses the original breach vector across three dimensions. Each control maps to a specific failure mode that existed prior to this implementation.

  • Code cannot be cloned locally. Blocked by GitHub IP Allowlist (only VPN/VDI IPs are permitted) combined with VDI lockdown — no local drive mapping means no filesystem the contractor controls.
  • Code cannot be pushed to a personal repository. Blocked at the identity layer (EMU accounts cannot interact with repos outside experity-emu) and at the network layer (SSH blocked, personal GitHub HTTPS denied, tenant restriction header enforced).
  • All activity is fully logged and alerted. Monitored end-to-end by Rapid7 InsightIDR (SIEM + UEBA) and CrowdStrike Falcon (endpoint + process behavior). No action — authorized or not — occurs outside of the audit trail.
